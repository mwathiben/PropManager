# Internationalization (i18n) Runbook

PropManager ships in **English (`en`)** and **Kiswahili (`sw`)**. This
runbook is the contributor's reference for keeping the localization
spine healthy — when to use which translation engine, the key-naming
convention, how to add a new locale end-to-end, the Swahili term
glossary, and the **honest scope** of what is translated today vs what
is still English.

Shipped by the Phase-24 [I18N + SWAHILI] audit cycle (2026-05-15) on
top of the Phase-23 a11y spine. Cross-reference
[`docs/runbooks/accessibility.md`](./accessibility.md) — the same
"document reality, not aspiration" principle applies here.

## The two-engine model

PropManager has **two translation engines** running side by side. Knowing
which one to call is a contributor must-know:

| Engine | Lives in | Used for | API |
|--------|----------|----------|-----|
| **Laravel `trans()` / `__()`** | `lang/{locale}/*.php` (PHP arrays) | Server-rendered content: validation errors, Mailable subjects + Markdown bodies, PDF strings, controller flash messages | `__('emails.subjects.invoice_sent', ['number' => $n])`, `trans('validation.required', ['attribute' => 'email'])` |
| **vue-i18n (`useI18n().t`)** | `lang/{locale}.json` (single flat JSON bundle) | Client-rendered content: nav labels, button text, page titles, in-page strings, form labels in Vue components | `const { t } = useI18n(); t('common.save'); t('format.relative.days_ago', { count: 5 }, 5)` |

**Rule of thumb**: if the string is rendered by PHP (Mailable, PDF,
flash), use `__()` against a PHP lang file. If it's rendered by Vue,
use `useI18n().t()` against the JSON bundle.

### Why two engines?

- Mailables, PDFs, and validation messages are server-rendered — they
  cannot reach vue-i18n. Laravel's translator is the only option.
- Vue components run in the browser — bundling the entire server-side
  PHP lang tree into the JS payload would balloon the bundle. vue-i18n
  reads a per-locale JSON bundle that we publish via
  `HandleInertiaRequests::share()['i18n']`, which only ships the keys
  the frontend actually uses.

### The locale-resolution chain

A request resolves to a single active locale via
`app/Http/Middleware/SetLocale.php`. Priority order:

1. `users.locale` (the authenticated user's stored preference)
2. `session('locale')` (a guest's runtime choice, or set by the
   `locale.update` endpoint before the user row is updated)
3. The request's `Accept-Language` header (best supported match)
4. `config('app.fallback_locale')` — the app default (`en`)

`SetLocale` calls both `App::setLocale($locale)` (for `__()`) **and**
`Carbon::setLocale($locale)` (for server-rendered dates). It's
registered in `bootstrap/app.php` **before**
`HandleInertiaRequests::class` so the Inertia share sees the resolved
locale.

`HandleInertiaRequests::share()` exposes three i18n props to every
Inertia visit:

- `locale` — the resolved locale string (e.g. `"sw"`)
- `availableLocales` — `['en' => 'English', 'sw' => 'Kiswahili']`
- `i18n` — the full JSON bundle for the active locale
  (`lang/{locale}.json`, falling back to `lang/{fallback}.json`)

The vue-i18n engine in `resources/js/app.js` consumes these on initial
paint and re-hydrates on every Inertia navigation, so a locale switch
re-renders the whole UI translated without a hard reload.

## Key-naming convention

**Dotted, namespaced, snake_case**.

| Namespace | Belongs in | Purpose | Example |
|-----------|------------|---------|---------|
| `common.*` | both | Cross-cutting nouns/verbs ("save", "cancel", "email") | `common.save` |
| `nav.*` | JSON | Sidebar nav labels, section dividers, skip-link, aria-labels | `nav.dashboard` |
| `menu.*` | JSON | User-menu items (Account / Support / Billing / Log Out) | `menu.log_out` |
| `role.*` | JSON | Role badges (landlord, caretaker, tenant) | `role.landlord` |
| `banner.*` | JSON | Impersonation + DPA-4 restriction banners | `banner.viewing_as` |
| `auth.{page}.*` | JSON | Auth pages (login, register, forgot, reset, confirm, verify, tfa) | `auth.login.submit` |
| `format.relative.*` | JSON | "Yesterday", "Today", "X days ago" — used by `useFormatters` | `format.relative.yesterday` |
| `empty.*` | JSON | Empty-state defaults | `empty.default_title` |
| `brand.*` | JSON | App-name + brand-string sr-only labels | `brand.go_to_dashboard` |
| `validation.*` | PHP (`lang/{locale}/validation.php`) | Laravel validation messages | `validation.required` |
| `emails.subjects.*` | PHP (`lang/{locale}/emails.php`) | Mailable subjects | `emails.subjects.invoice_sent` |
| `emails.{type}.*` | PHP (`lang/{locale}/emails.php`) | Mailable body strings (titles, CTA labels) | `emails.payment.details_title` |
| `pdfs.*` | PHP (`lang/{locale}/pdfs.php`) | Invoice/receipt PDF strings | `pdfs.invoice.heading` |
| `messages.*` | PHP (`lang/{locale}/messages.php`) | Generic server-side strings (flash messages, etc.) | `messages.profile.saved` |

**Placeholders use Laravel's `:token` syntax in PHP files and vue-i18n's
`{token}` syntax in JSON files.** The key-parity watchdog (Phase24CiTest)
will fail if a translated string's placeholder set drifts from the
English source.

## Adding a new locale end-to-end

For example, adding French (`fr`):

1. **Add the code to `config/app.php`**:
   ```php
   'available_locales' => [
       'en' => 'English',
       'sw' => 'Kiswahili',
       'fr' => 'Français',
   ],
   ```
2. **Copy the PHP lang tree**:
   ```bash
   cp -r lang/en lang/fr
   # then translate each file in lang/fr/*.php
   ```
3. **Copy the JSON bundle**:
   ```bash
   cp lang/en.json lang/fr.json
   # then translate every value, KEEPING THE KEY SET IDENTICAL
   ```
4. **Add a Carbon locale mapping in `useFormatters.ts`**:
   ```ts
   const INTL_LOCALES = {
       en: { number: 'en-KE', date: 'en-GB' },
       sw: { number: 'sw-KE', date: 'sw-KE' },
       fr: { number: 'fr-FR', date: 'fr-FR' },
   };
   ```
5. **Run the parity gate** (`php artisan test tests/Feature/I18n`). The
   Phase24CiTest watchdog asserts the new locale mirrors the English
   key set across PHP files + the JSON bundle, and that placeholder
   tokens match exactly. If anything is missing, the test names the
   exact key.

That's it — `SetLocale`, `HandleInertiaRequests`, the LocaleSelector
component, vue-i18n hydration, and Currency formatting all read from
the supported-locale config and pick up the new entry automatically.

## Translation review workflow

1. **English first**: the English string is the canonical source. Add
   it to `lang/en.json` (or the appropriate `lang/en/*.php` file) with
   the contributor's best phrasing. Aim for the user-facing string the
   way it should sound to a native English speaker.
2. **Translate to Swahili** (and any other supported locale) **in the
   same PR**: the Phase24CiTest key-parity watchdog will fail CI if the
   English key has no Swahili pair. Translation cannot lag behind the
   English source.
3. **Translator review**: a native or fluent Kiswahili speaker reviews
   the translation. Prefer natural Kenyan business Kiswahili — avoid
   Tanzanian formal Kiswahili calques that sound stiff in Nairobi.
4. **Glossary check**: domain terms must match the glossary above.
   Departures are flagged in the PR description.
5. **Phase24CiTest in CI**: shrink-only key-parity + placeholder
   parity. A PR that adds a new English key without a Swahili pair, or
   that drifts placeholder tokens between locales, fails CI.

## Honest scope

**What IS translated** (as of Phase-24):

- Validation messages (`lang/{locale}/validation.php`)
- Mailable subjects (15 user-facing Mailables) + Markdown bodies
- Invoice / receipt PDFs (`lang/{locale}/pdfs.php`)
- The **chrome**: `AuthenticatedLayout` (nav, user menu, banners,
  dividers), `GuestLayout`, all 7 auth pages (Login, Register,
  ForgotPassword, ResetPassword, ConfirmPassword, VerifyEmail,
  TwoFactorChallenge), `EmptyState` default title, the `LocaleSelector`
  itself, the relative-date helper (`format.relative.*` keys)
- Carbon-rendered server-side dates (via `Carbon::setLocale` from
  `SetLocale` middleware, and `->translatedFormat()` at user-facing
  render sites — invoice PDFs, mailable body dates, scheduler
  reminders, GDPR/consent dates)
- Currency + number formatting (Intl locale derived from active app
  locale; `Currency::localeFor($locale)` exposes a `sw-KE` branch for KES)

**What is still ENGLISH-ONLY** (and is NOT a "deferral" — it is the
explicit Phase-24 scope boundary):

- The long tail of per-page Vue strings: dashboard cards, finance
  pages, ticket UIs, settings tabs, admin tables, etc. Translating
  them is incremental post-Phase-24 work — file-by-file as features
  evolve, not a big-bang sweep. There are ~60+ pages.
- Console-command echo output and internal CSV report headers — these
  go to operators / landlords doing exports, not directly to tenants.
- Internal alert emails (`BreachReportedAlert`, `BreachEscalationAlert`,
  `FailedWebhookAlert`, `ReconciliationAlert`) — these go to ops, not
  tenants, and stay in English deliberately.

**The honest commit**: a Swahili user gets a translated auth flow, a
translated nav, translated invoices + PDFs + emails, translated dates +
numbers, and an English long-tail of in-page content. That's the line.
The CI key-parity gate guarantees we don't ship a half-translated key
set within scope — every English key in `lang/en.json` and `lang/en/*.php`
has a matching Swahili pair.

## CI gates

The Phase-24 watchdog suite (`tests/Feature/I18n/Phase24*Test.php`)
enforces the spine:

| Test | What it guards |
|------|----------------|
| `Phase24InfraTest` | locale column, supported_locale config, SetLocale middleware registration + resolution priority, Inertia share, locale-switch endpoint |
| `Phase24FrontTest` | vue-i18n is a dependency, useI18n composable + LocaleSelector exist, app.js wires the engine, chrome files use `t()`, chrome key-space present in every locale bundle, no custom date pickers (or they must use useFormatters) |
| `Phase24BackendTest` | User implements HasLocalePreference, subject key parity, Mailable subjects use `__()`, NotificationService passes the User model to Mail::to |
| `Phase24FormatTest` | useFormatters reads page.props.locale, relative phrases use i18n keys, `format.relative.*` keys present in every locale, Currency has a sw-KE branch, SetLocale calls Carbon::setLocale, user-facing dates use `->translatedFormat()` |
| `Phase24CiTest` | **KEY PARITY** — every supported locale's PHP lang files + JSON bundle have EXACTLY the English key set; placeholder tokens match between English and the translation |
| `Phase24SwahiliTest` | lang/sw/*.php files exist + mirror English; translator resolves Swahili |

These are cheap source-level assertions that catch regressions during
contributor review. They are NOT a substitute for human translator
review — they guarantee structural integrity, not translation quality.

## Phase 43 [I18N-DEPTH] additions

Phase-42 pre-cycle hot-fix `16cbafd` exposed a structural gap:
namespaced PHP lang files were silently invisible to the Vue
frontend because `getI18nBundle()` was only reading
`lang/{locale}.json`. The fix added the `lang/{locale}/*.php`
merge — Phase 43 closes the loop by making sure that class of
bug can't recur silently.

### `lang:coverage` watchdog (LANG-COVERAGE-1)

```
php artisan lang:coverage              # locale=en, dir=resources/js
php artisan lang:coverage --json       # machine-readable
php artisan lang:coverage --locale=sw  # check Swahili bundle
```

Walks `resources/js/**/*.{vue,ts,tsx,js,jsx}`,
regex-extracts every `$t('...')` / `t('...')` / `i18n.t('...')`
call with a literal key, and asserts each key resolves against
the same merged bundle `HandleInertiaRequests::getI18nBundle()`
ships to the client. Dynamic keys (`t('prefix.' + suffix)`) are
skipped — out of scope for static analysis.

`Phase43LangCoverageTest` runs this in CI; PR fails on any miss.

### Hardcoded English baseline (LANG-COVERAGE-2)

`App\Support\HardcodedEnglishScanner` counts hardcoded English
text nodes inside Vue `<template>` blocks.
`Phase43HardcodedStringBaselineTest::BASELINE = 3294` is the
**shrink-only ratchet** (Phase-22 NPlusOneBaseline pattern).
New code MUST use `$t()`. Migrating an existing literal lets you
ratchet the constant down. Failure message surfaces the top 5
offending files.

**To opt out** a brand name, proper noun, or intentionally-untranslated
string, place `<!-- i18n-ignore -->` on the line OR the previous line.

ESLint-side enforcement is a Phase-44 candidate — adds in-IDE
feedback so the PR-time scanner isn't the first place a dev
sees the violation.

### `lang:audit` cron (LANG-AUDIT-1)

Daily at 04:15 Africa/Nairobi. Diffs every locale's key set
against English, emits
`i18n_missing_keys_count{namespace,locale}` Prometheus gauge.
Sev3 alert at threshold 0 for pinned namespaces (`auth`,
`common`, `validation`, `payments`); sev4 for loose namespaces.

### `lang:check` CI gate (LANG-AUDIT-2)

Hard gate run in CI between phpstan + tests. `--pinned-only`
exits 1 with diff report on first missing key in pinned
namespaces. Loose-namespace drift is the gauge's job — `lang:check`
is the merge blocker.

### `lang:suggest` translation seeding (LANG-AUDIT-3)

```
php artisan lang:suggest payments --target=sw          # stub fills
php artisan lang:suggest payments --target=sw --driver=google
```

Emits a ready-to-paste PHP array fragment for the namespace's
missing keys in the target locale. Stub driver (default) returns
`[TODO:<locale>] <english>` placeholders for hand-translation;
Google + DeepL drivers fall back to Stub when the API key
(`config('i18n.google_api_key')` / `deepl_api_key`) is missing,
so dev + CI never accidentally make external calls.

### Adding a new translation key (Phase 43 contract)

1. Decide JSON (flat UI) vs PHP namespaced (domain string).
2. Add to en + sw simultaneously.
3. Run `php artisan lang:coverage` to confirm any `$t()` callsite
   in `resources/js` resolves.
4. Run `php artisan lang:audit` to confirm parity.
5. Phase24CiTest enforces parity at PR time.

### RTL support (Phase 44 — production)

`App\Support\LocaleHelper::isRtl(string $locale): bool` reads
`config('i18n.rtl_locales')` (default `['ar', 'he', 'fa', 'ur']`).
`app.blade.php` binds `dir="ltr|rtl"` on the html element.
`scripts/migrate-to-logical-properties.mjs` ran en masse in
Phase 44 RTL-MIGRATE-1 — Tailwind LTR utilities (`ml-`, `mr-`,
`pl-`, `pr-`, `left-`, `right-`, `border-l/r-`, `rounded-l/r-`,
`text-left|right`) are now logical properties (`ms-`, `me-`,
`ps-`, `pe-`, `start-`, `end-`, `border-s/e-`, `rounded-s/e-`,
`text-start|end`). The ESLint rule `propmanager/no-ltr-class`
catches new LTR residue at lint time; PHP-side, the Phase 44
codemod stays available for re-runs.

Arabic is a first-class locale: `config('app.available_locales')`
includes `'ar' => 'العربية'`; `lang/ar/*.php` mirrors `lang/en/`;
`lang/ar.json` mirrors `lang/en.json`. Many strings still carry
`[TODO-ar]` placeholders — the marker uses a hyphen (not colon)
to avoid colliding with Laravel's `:token` placeholder regex
that Phase24CiTest enforces. Run `php artisan lang:coverage` +
`php artisan lang:audit` to find the placeholders that still
need a translator pass; `common/brand/nav/menu/role` are
hand-translated.

The `useRtlAware` composable (`resources/js/composables/`)
exposes `isRtl`, `forwardKey`, `backwardKey`, and pipes
direction changes through `useAnnouncer` so screen readers
hear a polite `aria-live` notification when the layout flips.
Use `useRtlAware` in any component that handles arrow keys
(Modal, Carousel, Tabs, Dropdown) — do not hardcode
`ArrowLeft`/`ArrowRight`.

#### Adding another RTL locale

1. Add the code to `config('app.available_locales')` (e.g. `'he' => 'עברית'`).
2. The code is already in `config('i18n.rtl_locales')` for ar/he/fa/ur.
   Add it there if not (`RTL_LOCALES` set in `useRtlAware.ts` mirrors this).
3. Create `lang/<code>/` mirroring `lang/en/`; create `lang/<code>.json`.
   Use `[TODO-<code>]` placeholders (hyphen, not colon).
4. `php artisan test --filter='Phase24CiTest|Phase44I18nRtlSurfaceTest'`
   gates everything.
5. Add the locale to `playwright.rtl.config.ts` `locale:` if you want
   visual-regression coverage; otherwise the existing `ar` snapshots
   cover the layout contract.

#### Visual regression

`npm run test:rtl` — diff against committed snapshots in
`tests/a11y/rtl/__screenshots__/`.
`npm run test:rtl:update` — refresh baselines (commit the new
`.png` files in the same CL as the visual change).
Threshold: 1% pixel diff (`maxDiffPixelRatio: 0.01`). Layout
regressions blow past it; font sub-pixel noise stays under.

### Phase 43 common bugs + fixes

| Symptom | Cause | Fix |
|---------|-------|-----|
| Vue renders literal `'namespace.key'` | Key not in merged bundle | `php artisan lang:coverage` shows missing keys; add them |
| Phase24CiTest parity fail | en has keys sw doesn't (or vice versa) | `php artisan lang:audit` lists the drift |
| Phase43HardcodedStringBaselineTest fail | New literal slipped in | Wrap in `$t()` OR add `<!-- i18n-ignore -->` |
| All keys broken after a deploy | `getI18nBundle()` merge regression | Compare against Phase-42 `16cbafd`; restore namespace merge |

## Phase 52 — Auto-translate (driver-backed `lang:suggest`)

Owner: I18n workstream. Last touched: Phase 52.

Phase 43 shipped a TranslationSuggestionService scaffold whose `stub`,
`google`, and `deepl` "drivers" all fell through to the stub. Phase 52
wired the real Google + DeepL implementations behind a
`TranslationDriverInterface` + factory + cost guard. This section is the
operator-facing reference for the auto-translate workflow.

### Driver selection

| Driver  | Config key                  | Endpoint                                   | Per-char cost | When to use |
|---------|-----------------------------|--------------------------------------------|---------------|-------------|
| `stub`  | (default; no creds needed)  | n/a                                        | $0            | Local dev, CI, any flow that just needs `[TODO:locale]` placeholders |
| `google`| `I18N_GOOGLE_TRANSLATE_API_KEY` | `https://translation.googleapis.com/...` | ~$20 / M chars | High volume, locales DeepL does not support, fast iteration |
| `deepl` | `I18N_DEEPL_API_KEY`        | auto: api-free.deepl.com OR api.deepl.com | ~$25 / M chars (pro) | European languages + Arabic where DeepL quality outperforms Google |

### API key setup

Add to `.env`:
```
I18N_SUGGESTION_DRIVER=deepl
I18N_DEEPL_API_KEY=<your-key>
I18N_DEEPL_FORMALITY=formal      # optional; de/fr/es/it/nl/pl/pt/ja/ru only
I18N_DEEPL_GLOSSARY_ID=<glossary># optional
```

If the DeepL key ends in ':fx' the driver auto-routes to the free tier
(`api-free.deepl.com`); otherwise it uses pro. Same env var for both.

### Budget guard

Every non-stub driver is wrapped in `CostAwareDriver`. Before each call:

1. `TranslationCostTracker::canSpend(estimatedUsd)` checks the rolling
   24h spend against `config(\047i18n.daily_budget_usd\047, 20.0)`.
2. If past budget, the call falls back to the stub silently (logged).
3. On a successful response, the actual cost is recorded into the
   `i18n:translation:spend:daily:total:<date>` cache key + a per-locale
   sibling key.

The Phase-52 sev3 alert in `alert-thresholds.md` fires at the same
threshold so ops can decide before the hard cap.

### `--apply` workflow

```
# Print suggestions to stdout (default; safe to inspect)
php artisan lang:suggest accounting --target=ar --driver=deepl

# Atomically write suggestions into lang/ar/accounting.php
# Creates lang/ar/accounting.php.bak.<timestamp> first; never overwrites existing keys.
php artisan lang:suggest accounting --target=ar --driver=deepl --apply

# List every row still flagged with [TODO:<locale>] or [TODO-<locale>]
php artisan lang:review --locale=ar --namespace=accounting
```

`LangFileWriter::merge` is additive: existing translations are NEVER
overwritten by a `--apply` run. The merge is recursive (deep) so
partial namespace fills compose correctly.

### RTL post-translation review

Google's MT for Arabic occasionally swaps RTL/LTR markers; DeepL is
usually cleaner. After a large `--apply` run, eyeball the highest-
traffic namespaces (`auth`, `common`, `payments`) in a browser with
the locale set to  to spot any direction glitches the Phase-44
RTL snapshot suite did not catch.

### Incident playbook: runaway spend

1. Alert fires: `i18n_translation_spend_usd_24h` > $20/day.
2. Tail `storage/logs/laravel.log` for `TranslationCostTracker` /
   `CostAwareDriver falling back` entries — confirms the budget cap
   was hit.
3. Identify the running `lang:suggest --apply` (shell history /
   `ps aux | grep artisan`); kill it.
4. Revert affected lang files from the `.bak.<timestamp>` snapshots:
   `cp lang/ar/accounting.php.bak.1234567890 lang/ar/accounting.php`.
5. Investigate root cause: was the call legit (raise
   `I18N_DAILY_BUDGET_USD`) or a loop bug (file an issue)?

### Phase 43 → Phase 52 deferred-finding lineage

Phase 43 explicitly shipped the stub-only driver pattern. The
`TranslationSuggestionService` source carried the comment "Real
implementation deferred. Phase 44+ wires the actual API once the
workflow proves out via Stub." That deferral closed in Phase 52: the
driver pattern + factory + cost guard + apply workflow all landed.
The comment is removed from the service source; new contributors
arriving at the file see the shipping interface, not the historical
caveat.

Future i18n cycles can extend by adding new driver classes that
implement `TranslationDriverInterface`; the factory needs a single
new match arm + a config-key for the API credential. Don't touch
caller code.
